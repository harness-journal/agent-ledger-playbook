Chapter 1

왜 AI 작업은 기록이 필요해지는가

AI는 대화창에서 쓸 때 가장 편하다. 문제는 편한 만큼 흔적이 약하다는 점이다. 좋은 결과만 복사해 가면 작업의 이유와 과정은 사라진다.

좋은 답변은 남지만, 좋은 작업은 남지 않는다

대부분의 사용자는 AI를 "그때그때 답해주는 도구"로 쓴다. 그래서 최종 결과만 저장하고, 그 결과가 어떻게 나왔는지는 저장하지 않는다. 그런데 실제 업무에서는 최종 결과보다 과정이 더 중요할 때가 많다.

예를 들어, 글 초안이 이상하게 나왔을 때 원인을 찾으려면 최소한 아래를 다시 볼 수 있어야 한다.

  • 내가 정확히 무엇을 요청했는가
  • AI가 어떤 식으로 해석했는가
  • 중간에 어떤 도구나 검색을 사용했는가
  • 어떤 모델을 썼는가
  • 누가 그 작업을 실행했는가

설명 가능한 작업이 결국 더 강하다

AI 사용은 점점 더 많은 조직에서 일상 업무가 되고 있다. 이때 중요한 것은 "한 번 잘 나온 결과"보다 "왜 이런 결과가 나왔는지 다시 설명할 수 있는 작업"이다. 설명 가능해야 수정이 가능하고, 수정 가능해야 팀의 품질이 올라간다.

핵심 문장 AI에게 일을 시켰다면, 결과만 남기지 말고 무슨 일이 일어났는지 나중에 다시 볼 수 있게 만들어라.
Chapter 1 기억에 의존하는 AI 사용은 실력보다 우연에 가까워진다.

Chapter 2

저장하지 않으면 생기는 7가지 문제

Problem 01
같은 실수를 반복한다

이전 실패가 어떻게 만들어졌는지 모르기 때문에, 비슷한 작업에서 같은 오류가 다시 난다.

Problem 02
좋은 프롬프트가 자산이 안 된다

잘 먹힌 요청을 다시 재사용하지 못한다. 결국 성과가 개인 기억에만 남는다.

Problem 03
결과 품질을 비교할 수 없다

어떤 모델, 어떤 지시, 어떤 도구 조합이 좋았는지 비교할 수 없다.

Problem 04
비용이 새는 지점을 모른다

무슨 작업에 어떤 모델을 얼마나 썼는지 모르면 최적화가 불가능하다.

Problem 05
팀 학습이 누락된다

개인이 깨달은 좋은 사용법이 팀 규칙으로 번역되지 않는다.

Problem 06
책임 소재가 흐려진다

누가 실행했는지, 누가 승인했는지 남지 않으면 운영이 취약해진다.

Problem 07
보안 사고를 알기 늦어진다

반대로 아무 기준 없이 전부 저장하면 민감정보 사고가 날 수 있다. 그래서 저장 시스템은 기록성과 안전성을 함께 가져야 한다.

Chapter 2 기록이 없으면 학습이 누적되지 않고, 기록이 무질서하면 보안이 깨진다.

Chapter 3

최소한 무엇을 저장해야 하는가

처음부터 다 저장할 필요는 없다. 하지만 너무 적게 저장하면 나중에 원인을 다시 볼 수 없다. 이 플레이북은 최소 필수 5가지를 제안한다.

Required 01
Prompt

내가 뭐라고 시켰는지. 나중에 결과가 이상할 때 가장 먼저 다시 봐야 하는 항목이다.

Required 02
Response

AI가 뭐라고 답했는지. 결과뿐 아니라 판단 흐름의 흔적이다.

Required 03
Tools

중간에 검색, 파일 읽기, 테스트 실행 같은 도구를 썼는지.

Required 04
Actor

누가 실행했는지. 개인 작업자라면 자신, 팀이라면 정확한 주체를 남긴다.

Required 05
Model

어떤 모델을 썼는지. 품질 비교와 비용 추적을 위해 필수다.

Recommended
Timestamp, Next Action, Artifact, Approval

시간, 다음 액션, 결과 파일, 승인 여부까지 남기면 운영 품질이 훨씬 좋아진다.

실무 기준 최소 저장 기준은 "뭘 시켰나, 뭐가 나왔나, 중간에 뭘 했나, 누가 실행했나, 어떤 모델이었나"의 다섯 질문에 답할 수 있어야 한다.
Chapter 3 이 다섯 가지가 쌓이면 그것이 개인 또는 팀의 AI 작업 자산이 된다.

비개발자용 저장 방식

비개발자라고 해서 저장 시스템을 못 만드는 것은 아니다. 오히려 가장 좋은 시작은 복잡한 도구가 아니라 가볍게 계속 남기는 방식이다.

방법 좋은 점 한계 언제 적합한가
메모장 오늘 바로 시작 가능 검색과 구조화가 약함 개인 실험 초기
Markdown work log 가볍고 버전 관리 가능 분석 자동화는 약함 개인 작업자, 크리에이터
스프레드시트 필드별 비교와 정렬이 쉬움 긴 프롬프트/응답 관리가 번거로움 운영자, PM, 비개발 팀

- tools: web_search
- next_action: 게시용 최종 다듬기

Chapter 5

개발자용 저장 방식

개발자는 파일 기반 로그에서 시작해도 좋지만, 곧 구조화된 저장과 조회의 필요를 느끼게 된다. 이때 중요한 것은 멋진 시스템보다 단계적으로 올리는 것이다.

JSON은 자동화 가능한 최소 구조다

Markdown이 사람에게 좋다면 JSON은 도구에 좋다. CLI, 자동화 스크립트, 후속 분석, 대시보드 연결이 쉬워진다. 

{
  "timestamp": "2026-04-02T15:00:00",
  "task": "summarize_simon_april_posts",
  "actor": "inabe",
  "model": "gpt-5",
  "prompt": "Simon 4월 글 핵심 정리",
  "response": "운영, 추적, 권한 중심",
  "tools": ["web_search"],
  "next_action": "write_threads_draft"
}

SQLite는 "다시 보기"를 가능하게 한다

JSON이 기록이라면 SQLite는 조회다. 작업별, 모델별, 사람별, 실패 패턴별로 다시 묻고 비교하려면 결국 queryable storage가 필요해진다.

바로 이 지점에서 Simon Willison의 방식이 강해진다. 그는 대화창을 저장하는 데서 멈추지 않고, 저장된 것을 다시 조회하고, plugin으로 usage와 actor까지 구조화한다.

Chapter 5 개발자에게 중요한 건 저장 형식보다 조회 가능한 구조다.

가장 쉬운 시작법

대부분의 사람은 시작부터 너무 큰 시스템을 만들려고 한다. 그러면 아무것도 남기지 못한다. 이 플레이북은 가장 쉬운 5줄 로그부터 시작할 것을 권한다.

  1. 작업 이름을 쓴다
  2. 프롬프트를 적는다
  3. 응답을 적는다
  4. 도구와 모델을 적는다
  5. 다음 액션을 적는다
좋은 시작의 기준 가장 쉬운 시스템이 가장 좋은 시스템이다. 나중에 SQLite로 갈 수 있더라도, 오늘은 Markdown 파일 하나로 시작해도 충분하다.

Chapter 7

Simon Willison은 어떻게 저장 시스템을 만들고 있나

이 장은 "사이먼이 대단하다"를 말하는 장이 아니다. 공개된 글과 릴리스에서 실제로 확인되는 저장 워크플로를 추려서, 우리가 복제 가능한 구조로 읽는 장이다.

1. LLM 실행을 대화창에서 끝내지 않는다

Simon의 llm 도구는 prompt와 response를 SQLite에 저장하는 방향을 택한다. 출발점부터 "채팅한 뒤 잊어버리는 사용법"이 아니라 "실행과 결과를 데이터로 남기는 사용법"이다.

2. 저장한 뒤 다시 조회한다

저장은 끝이 아니다. Simon은 저장된 SQLite를 Datasette로 다시 열어 보는 흐름을 제시한다. 이 구조는 중요하다. 로그가 있어도 다시 보기 어렵다면 학습으로 이어지지 않기 때문이다.

3. 2026년 4월에는 저장 범위를 더 넓혔다

4월 1일 공개 릴리스에서는 tool call loops, prompts/responses/tool calls logging, actor, allowed/default model, usage 기록 같은 항목이 반복해서 보인다. 즉, 단순 대화 저장에서 운영 로그 저장으로 계속 확장하고 있다.

Simon의 공개 워크플로를 한 줄로 줄이면 LLM 실행 → SQLite에 저장 → Datasette로 조회 → plugin으로 usage, tool calls, actor, model policy를 점점 더 구조화
Chapter 7 Simon은 "더 똑똑한 모델"보다 "더 관리 가능한 AI 시스템"에 가까운 방향을 보여준다.

Chapter 8

Simon 방법론을 그대로 따라 할 때의 최소 구현

Level 1
파일 로그 버전

작업마다 Markdown 블록을 쌓는다. 비개발자와 개인 작업자에게 가장 좋은 시작점이다.

Level 2
JSON 버전

자동화와 구조화가 필요하면 JSON으로 옮긴다. 나중에 DB로 가기 쉬워진다.

Level 3
SQLite 버전

모델별, 사람별, 작업별로 다시 묻고 비교해야 할 때 올린다.

Level 4
조회 화면

저장만 하지 않고 다시 보는 화면을 붙인다. 그래야 운영과 리뷰가 가능해진다.

이 플레이북이 Simon을 적극 활용하는 이유는 여기에 있다. Simon은 저장을 "기록 습관"으로만 보지 않고, 저장과 조회를 하나의 운영 루프로 만든다. 이 점이 실무적으로 매우 강하다.

Chapter 8 가장 중요한 것은 복잡한 시스템이 아니라 점진적 확장 가능성이다.

Mitchell Hashimoto 관점에서 보면 왜 이게 하네스 엔지니어링인가

Simon이 보여주는 것은 저장과 조회의 구조다. Mitchell Hashimoto의 언어로 바꾸면, 이것은 결국 에이전트가 같은 실수를 반복하지 않게 만드는 하네스의 일부다.

기록은 반복 실수를 줄이기 위한 전제다

기록이 없으면 같은 실수를 다시 보지 못하고, 다시 보지 못하면 규칙으로 바꾸지 못한다. Hashimoto가 강조하는 하네스는 결국 이런 반복 실수 방지 구조와 맞닿아 있다.

AGENTS.md와 로그는 서로 연결된다

AGENTS.md는 "앞으로 어떻게 일할 것인가"를 정의하고, 로그는 "실제로 어떻게 일했는가"를 남긴다. 이 둘이 분리되면 규칙은 이상적 선언으로 남고, 연결되면 실제 운영 기준이 된다.

Chapter 9 좋은 하네스는 지시문만이 아니라, 그 지시가 현실에서 어떻게 작동했는지의 기록까지 포함한다.

무엇을 저장하면 안 되는가

Do Not Store
시크릿, 토큰, 비밀번호

가장 기본이다. 자동 저장 시스템에 들어가면 안 되는 1순위다.

Do Not Store
고객 원문 민감정보

필요하면 요약본 또는 마스킹된 형태만 남긴다.

Do Not Store
내부 비공개 자산 전체

필요한 필드만 추려 저장해야 한다.

Do Not Store
민감정보가 섞인 전체 프롬프트

가능하면 요약 또는 redacted prompt를 사용한다.

로그를 남기되 위험하지 않게 만드는 법

  • 기본은 local-first로 시작한다
  • 민감정보는 저장 전 redaction을 먼저 고려한다
  • 프로젝트별 저장 위치를 분리한다
