# Review And Feedback

이 문서는 저장된 로그를 다시 보는 방법을 정의합니다.

저장은 시작일 뿐입니다.
리뷰가 없으면 로그는 파일이고, 리뷰가 있으면 운영 자산이 됩니다.

## Main Rule

매주 최소 한 번은 아래 질문에 답할 수 있어야 합니다.

1. 어떤 작업이 가장 많이 반복됐는가
2. 어떤 프롬프트가 가장 잘 작동했는가
3. 어떤 실패가 가장 자주 나왔는가
4. 어떤 모델이 비용 대비 가장 적절했는가
5. 다음 주에 무엇을 바꿀 것인가

## Weekly Review Flow

### Step 1. Collect

한 주 동안 남긴 로그를 모읍니다.

대상:

- work-log.md
- ai-task.json
- artifact link
- approval note

### Step 2. Cluster

비슷한 작업을 묶습니다.

예:

- 리서치
- 글쓰기
- 코드 작성
- 요약
- 분류

### Step 3. Extract

아래를 추출합니다.

- 가장 유용했던 프롬프트
- 가장 자주 실패한 패턴
- 자주 쓰는 모델
- 불필요하게 비싼 작업
- 다음 주 규칙 후보

### Step 4. Turn Into Rules

Hashimoto 관점에서 중요한 건 여기입니다.

반복 실수를 그냥 회고로 끝내지 말고, 다음 주 규칙으로 바꿔야 합니다.

예:

- "긴 글 요약은 먼저 목차를 뽑게 한다"
- "코드 수정은 반드시 테스트 실행 여부를 tool field에 남긴다"
- "외부 자료 사용 시 exact date를 남긴다"

## Failure Pattern Prompts

AI에게 리뷰를 맡길 때 쓸 수 있는 질문:

- 어떤 실패 패턴이 가장 자주 반복됐나
- 어떤 작업에서 prompt가 너무 모호했나
- 어떤 모델은 비용 대비 과했나
- 어떤 작업은 JSON 저장으로 올리는 게 맞나

## Review Outputs

최소 산출물은 아래 3개입니다.

1. 이번 주 반복 실수 3개
2. 다음 주에 바꿀 규칙 3개
3. 계속 유지할 좋은 패턴 3개

## Anti-Patterns

- 로그는 남기지만 다시 보지 않는다
- 실패를 발견해도 규칙을 안 바꾼다
- 개인 회고만 있고 팀 규칙으로 번역하지 않는다
- 모델 비용을 전혀 보지 않는다
